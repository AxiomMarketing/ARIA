## CRITICAL: ARIA COMMENT SYNTAX

The ARIA lexer handles dashes as follows:

| Syntax | What it is | What happens |
|--------|-----------|--------------|
| `--` text | Line comment | **COMPLETELY DISCARDED** by lexer â€” never reaches parser |
| `---` text | Doc comment | **PRESERVED** in AST and generated output |
| `->` | Arrow operator | Used in behavior transitions |

This means:

```aria
# âťŚ WRONG â€” these lines are INVISIBLE to the compiler:
requires
  -- player must be alive
  -- player must have role chasseur

# âś… CORRECT â€” formal expressions the compiler can parse:
requires
  player.isAlive == true
  player.role == "chasseur"

# âś… CORRECT â€” doc comment + formal expression when prose context helps:

  --- La SorciĂ¨re ne peut utiliser la potion de vie qu'une seule fois
  sorciere.potionVieUsed == false
```

## EXPRESSION SYNTAX REFERENCE

Every line under `requires` and `ensures` must be a valid ARIA expression. The expression parser supports:

| Category | Operators | Examples |
|----------|-----------|----------|
| Comparison | `==`, `!=`, `<`, `<=`, `>`, `>=` | `player.health > 0` |
| Logic | `and`, `or`, `not` (also `&&`, `\|\|`, `!`) | `a > 0 and b > 0` |
| Arithmetic | `+`, `-`, `*`, `/` | `old(balance) - amount` |
| Member access | `.` | `player.role`, `game.phase.name` |
| Function calls | `name(args)` | `old(x)`, `length(items)`, `contains(list, item)` |
| Literals | numbers, strings, booleans | `42`, `"active"`, `true` |
| Grouping | `(`, `)` | `(a + b) * c` |

**Special keywords:**
- `old(x)` â€” value of x before execution (use in `ensures`)
- `result` â€” return value of the contract (use in `ensures`)
- `self` â€” current value (use in type `where` clauses only)

### 2. Enrichment loop (for each spec)

For each skeleton:

#### a. Read the source files

- Read the source file mentioned in the header
- Read any corresponding test files (`.test.ts`, `.spec.ts`, `__tests__/`)
- Read JSDoc comments above each function

#### b. Replace TODOs and prose with formal expressions

For each contract in the spec:

**requires** â€” Derive formal expressions from:
- Early returns: `if (x < 0) return ...` â†’ `x >= 0`
- Throws on invalid input: `if (!email) throw ...` â†’ `email != ""`
- Assertions: `assert(amount > 0)` â†’ `amount > 0`
- Validation calls: `schema.parse(input)` â†’ mirror each constraint as expression
- Type guards: `if (typeof x !== 'number') throw` â†’ `x >= 0` (adapt to context)
- Null checks: `if (!user) throw ...` â†’ `user != null`
- Enum checks: `if (status !== 'active') throw ...` â†’ `status == "active"`

**ensures** â€” Derive formal expressions from:
- Return type: `function f(): { id: string }` â†’ `result.id != ""`
- Final returns: `return { ...input, status: "active" }` â†’ `result.status == "active"`
- State changes: `user.active = true; return user` â†’ `result.active == true`
- Value comparisons: `return balance - amount` â†’ `result == old(balance) - amount`

**on_failure** â€” Derive from:
- Detected throws: refine the `when` condition with a formal expression
- `if (balance < amount) throw new InsufficientFunds(...)` â†’ `when balance < amount`

**examples** â€” Derive from:
- Test files: extract `it('should ...', () => {...})` test cases
- For each test, extract input fixture and expected result
- Convert to ARIA `given/then` format with concrete values

**Advanced features** â€” When the source code reveals these patterns, use ARIA's advanced features:

| Source code pattern | ARIA feature to use |
|---|---|
| Generic return types (`Promise<Result<T>>`) | `type Result of T, E is Record` |
| Computed/derived values (`get total() { return sub + tax }`) | `total: Money computed as subtotal + tax` |
| Switch/dispatch on type field | `dispatch on field when "x" -> ContractX` |
| Rate limiting middleware | `rate_limit max N per interval per key` |
| Timeout wrappers (`Promise.race`) | `timeout N seconds` |
| Retry logic (exponential backoff) | `retry max N backoff exponential` |
| External service calls (Stripe, DB) | `depends_on StripeAPI, Database` |
| Multi-step transactions with rollback | `steps` + `compensate` (saga pattern) |
| State machines (enums + transitions) | `behavior` with `invariants` using `always`, `never`, `eventually` |
