# Step RV-03 тАФ Enrich Skeletons via Source Code

## MANDATORY EXECUTION RULES

- ЁЯЫС NEVER fabricate requires/ensures тАФ derive them from the actual source code
- ЁЯЫС NEVER touch `types` or `contracts` structure тАФ only fill in the TODO sections
- ЁЯЫС NEVER skip validation тАФ `aria check` must pass after each enrichment
- тЬЕ ALWAYS read the corresponding `.ts` source AND any `.test.ts` for context
- тЬЕ ALWAYS preserve the existing skeleton structure
- ЁЯУЛ YOU ARE AN ENRICHER, not a re-author

## CONTEXT BOUNDARIES

- Coming from: `step-rv-02-import.md` with skeletons in `specs/`
- Going to: `step-rv-04-drift.md` with enriched specs

## YOUR TASK

For each `.aria` skeleton in `specs/`, read the corresponding source `.ts` file (mentioned in the skeleton's header comment) and any matching test files, then fill in the `requires`, `ensures`, `examples`, and `on_failure` TODOs with content derived from the actual code.

---

## EXECUTION SEQUENCE

### 1. List skeletons to enrich

```bash
ls specs/*.aria
```

For each spec, the header comment looks like:

```aria
-- Generated by `aria import` from src/payment/charge.ts
```

Extract the source path. If the source no longer exists, skip with a warning.

### 2. Enrichment loop (for each spec)

For each skeleton:

#### a. Read the source files

- Read the `.ts` source mentioned in the header
- Read any corresponding `.test.ts` or `.spec.ts` in the same directory or `__tests__/`
- Read JSDoc comments above each function

#### b. Fill in TODOs

For each contract in the spec:

**requires** тАФ Derive from:
- Early returns: `if (x < 0) return ...` тЖТ `requires x >= 0`
- Throws on invalid input: `if (!email) throw ...` тЖТ `requires email exists`
- Assertions: `assert(amount > 0)` тЖТ `requires amount > 0`
- Validation calls: `schema.parse(input)` тЖТ mirror the schema constraints
- Type guards: `if (typeof x !== 'number') throw` тЖТ `requires x is Integer`

**ensures** тАФ Derive from:
- Return type: `function f(): { id: string }` тЖТ `ensures result.id exists`
- Final returns: `return { ...input, status: "active" }` тЖТ `ensures result.status == active`
- Side effects: `db.insert(x); return x;` тЖТ `ensures result.id exists` (after insert)

**on_failure** тАФ Derive from:
- Detected throws (already partially in skeleton): refine the `when` condition
- `if (validation fails) throw new ValidationError("X")` тЖТ `when X then return ValidationError`

**examples** тАФ Derive from:
- Test files: extract `it('should ...', () => {...})` test cases
- For each test, extract input fixture and expected result
- Convert to ARIA `given/then` format

#### c. Validate

```bash
npx aria-lang check specs/{file}.aria
```

If validation fails, fix the syntax error and re-validate.

#### d. Report progress

```
[3/12] specs/charge.aria тАФ enriched (4 contracts, 8 examples added)
```

### 3. Final summary

```
тЬУ Enriched {N} specs
  - {N_with_examples} now have examples
  - {N_with_requires} now have non-empty requires
  - {N_with_ensures} now have non-empty ensures
  - {N_with_failures} now have on_failure clauses

тЪа {M} specs had source files missing or unreadable (skipped)
```
